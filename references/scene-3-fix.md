# 场景 3：问题修复

适用信号："修复"/"bug"/"问题"/"报错"/"fix"/"error"/"异常"/"崩了"

通用八步闭环、执行卡、验证阶梯见 SKILL.md。落盘见 [output-formats.md](output-formats.md) 的 S3 模板。

## 场景独有要点

- **Route 是关键步**：先定位根因再动手，选最窄修复点。优先单文件/单函数/单配置项；避免重构、批量、跨模块联动；允许同一合同下代码+测试+文档一起改。
- 最小改动原则：能改 1 行不改 10 行。
- 根因修复，不在下游贴补丁。
- Verify 按问题类型选最窄验证：代码 bug → 测试/dry-run；配置错误 → 读回；数据问题 → 查询确认。

## 根因分析

```text
现象 What → 在哪 Where → 何时 When → 为什么 Why（根因在此）→ 怎么修 How（Route 依据）
```

## Prompt 模板

```markdown
你是修复工程师，用 Loop Mode 修复以下问题。

问题：[具体描述]
成功标准：[可量化判定，如"查询 < 100ms，现有数据无丢失"]
约束：最小改动；先 targeted 验证必要时升级；根因修复；完成后落盘。
禁止：未定位根因就动手；多个无关根因混进一轮（同一根因可代码+测试+文档同步）；改后不验证；用正式生产/全链路重跑证明局部修复（除非用户明确要求或 targeted 不足）。
```
